## 注册赠送逻辑的漏洞

### 关于漏洞的说明

我发现重复注册赠送代币的逻辑居然纯靠前端实现，这意味着任何人都能构建合约示例并且依靠这个恶意刷币。   
为了论证这一点，我构建了一个**重复注册攻击**的测试用例，用例中用到的 `MPScontractABI` 和 `MPScontractAddress` 都是公开信息。    

其中 `MPScontractAddress` 任何人都可以通过区块浏览器（如 Etherscan）看到；  
`MPScontractABI` 可以通过反编译字节码或查看源码获取 ABI。只要合约部署在链上，并且用户知道地址，就总有办法猜出接口，ABI 可以通过多种方式还原（如 Etherscan、Hardhat decode、bytecode 反推等）  

### 攻击步骤

在浏览器控制台引入 `Web3.js`：  

```js
var script = document.createElement('script');
script.src = 'https://cdn.jsdelivr.net/npm/web3@1.10.0/dist/web3.min.js';
document.head.appendChild(script);
```

### 已经论证无效的一些思路

- 思路：  
  在智能合约里限制每个哈希地址只能领取一次代币  
- 说明：  
  没用的。用户完全可以自己构建一个脚本，批量注册账户，然后再把 MPS 转进同一个账户。而且即便能解决注册送币的问题，也解决不了法币转换的问题。  

### 个人的建议

纯靠前端完成发币，理论上应该没有什么安全的办法（我目前想不到）。如果能在后端配置管理员私钥，然后带着私钥通过 API 请求区块链发币，应该是最稳妥的办法（但是目前好像只有前端能和 Metamask 交互？  

## 注册邮箱验证码的逻辑漏洞

用户注册的时候，邮箱验证码逻辑居然也是完全没用到后端，用户能通过各种方法破解或者绕过验证码来注册。  

### 建议

验证码的生成、校验应当都交给后端完成，前端只接受成功与否的响应
（已经分配 李昌泉 修复邮箱验证码的漏洞）
